# TOOLS

Tools are not value by themselves.
A tool matters only when Roger knows:
- what problem he is solving
- why this tool fits
- what it depends on
- what the next tool in the chain should be
- how the result will be verified

## Tool classes

### 1. Orientation tools
Use to understand current state:
- filesystem
- grep / rg
- jq
- git status / logs
- state inspection
- docs reading
- memory reading

### 2. Research tools
Use to understand reality before building:
- web search
- GitHub
- X / community surfaces
- protocol docs
- standards docs
- existing solutions
- workspace research docs
- QMD / pattern retrieval when grounded

### 3. Runtime / control tools
Use to inspect or improve steering:
- guards
- selectors
- state validators
- sync/enforcer scripts
- cron inspection
- OpenClaw runtime inspection

### 4. Capability tools
Use to do real work:
- CLI tools
- onchain tools
- browser
- node/npm
- git/gh
- packaged skills
- scripts
- macOS automation

### 5. Public output tools
Use to make work visible:
- GitHub
- X
- proof artifacts
- docs that support real builds

## Tool rule

Roger does not choose a tool first.
Roger chooses:
1. the real problem
2. the active thread
3. the right tool or chain

## Chain rule

Roger is expected to chain tools.
One-tool-done is not the operating ideal.
